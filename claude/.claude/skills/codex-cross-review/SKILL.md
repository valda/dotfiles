# codex-cross-review

Codex CLI（OpenAI）にクロスレビューを依頼するスキル。Claude 自身の作業に対するセカンドオピニオン用途。

レビュー以外の相談・バグ調査・分析・提案は対象外。

## 実行モード

デフォルトは **モード3（収束ループ）**。明示的な単発・再レビュー指示があるときだけ別モードを選ぶ。

判定優先順:

1. 「再レビュー」「resume」「更新したからもう一回見て」 → モード2
2. 「1 回だけ」「単発」「とりあえず見て」 → モード1
3. それ以外 → **モード3（収束ループ）**

Codex の指摘は 1 回で尽きるとは限らないため、収束まで回すのが既定の安全側。

### モード1: 単発レビュー

```bash
timeout 600 codex exec --sandbox read-only --cd <project_dir> - <<'PROMPT' 2>&1 | tail -120
<プロンプト本文>
PROMPT
```

- **`--full-auto` は使わない**: `--sandbox read-only` を黙って `workspace-write` に上書きする。`codex exec` は元々 `approval: never` の非対話実行なので `--sandbox read-only` だけでよい。
- バナーの `sandbox: read-only` と `session id: <uuid>` を確認する。session id は再レビューで使うので控える。

### モード2: 再レビュー（resume）

直前の Codex セッションのコンテキストを引き継ぐ:

```bash
timeout 600 codex exec resume <SESSION_ID> - <<'PROMPT' 2>&1 | tail -120
<プロンプト本文>
PROMPT
```

- **`--last` より session id 指定を優先する**: `--last` はマシン全体で最後の codex セッションを拾うため、別プロジェクトや codex-yolo-implement の実装セッションが挟まると取り違える。直前に他の codex 実行がないと確信できる場合のみ `--last` 可。
- resume サブコマンドは `--cd` / `--sandbox` を受け付けない（unexpected argument エラー）。sandbox 等は元セッションから引き継がれる。設定を変えたい場合は `-c key=value` で。

### モード3: 収束ループ（デフォルト）

重要指摘がゼロになるまで繰り返す。

1. モード1 と同形のコマンドで新規レビュー実行（**session id を必ず控える**）
2. 結果を「結果のトリアージ」基準でフィルタ
3. **スコープ膨張ガード**: 残った提案を採用する前に、各提案について判定:
   - 元 spec の「含まないもの」セクションを実質的に巻き戻すか？
   - 新規ファイル / 新規抽象化 / 新規設定キーを追加するか？
   - 観測されたバグではなく仮説的な失敗モードへの先回り対処か？

   いずれかに該当する提案は **自動採用しない**。ユーザーに採否を確認する。
4. **採用提案の裏取り**: 並行性・状態遷移・ジョブのライフサイクルに触れる採用提案は、新しいリグレッションを持ち込み得る。採用後は regression テストか実走で独立に検証する。codex の OK は検証の代わりにならない。
5. 残った重要指摘を報告 → 修正
6. 修正後、モード2（resume）で再レビュー
7. 終了条件は **両方** 満たすこと:
   - 重要指摘がゼロ
   - 直近イテレーションで新規ファイル / 新規概念の追加なし（または追加分はユーザー承認済み）

初回レビューで指摘ゼロが返り、トリアージ後も残件なしなら、モード2 を呼ばずに収束として報告してよい。

## プロンプトの渡し方（hang 回避）

インライン二重引用符は事故りやすい。plan doc 引用にバッククォートや `$()` が含まれると bash のコマンド置換で壊れる。引数渡しのつもりが stdin 待ちに入って無音ハングすることもある（`Reading additional input from stdin...` のまま CPU 0% で停止）。

推奨は `-` + quoted heredoc で stdin から渡す:

```bash
codex exec --sandbox read-only --cd <dir> - <<'PROMPT'
（バッククォートも $ もそのまま書ける）
PROMPT
```

長大なプロンプトはファイル経由でもよい: `codex exec ... - < /tmp/codex-prompt.md`

- stdin を渡している状態での `Reading additional input from stdin...` は正常ログ。stdin を渡していないのに出て止まったらハング → kill して stdin 方式で再投入。
- ハング判定: 出力ログのサイズが成長しているか + プロセスの CPU 時間（`ps -o pid,etime,time`）。ログ成長中は時間がかかっているだけ。

## 実行時間と待ち方

codex は起動に数十秒、レビュー完了まで数分〜10 分。

- フォアグラウンド: `timeout 600`（複雑な diff は 900）+ Bash の timeout を 600000〜900000ms。
- バックグラウンドは stdout 取りこぼしが起きうる。`--output-last-message <FILE>`（`-o`）で最終レスポンスをファイルに確定保存するか、`> /tmp/codex-out.log 2>&1` でリダイレクトする。短〜中規模ならフォアグラウンド + `tail` が確実。

## config.toml 起因の起動失敗

ユーザーの `~/.codex/config.toml` に現行 CLI が認識しないキー（例: `service_tier = "default"`）があると起動が失敗する。**ユーザーの config は編集せず**、`-c key=value` で CLI override する（例: `-c service_tier="flex"`）。

## プロンプトのルール

codex に渡すリクエストには以下の指示をすべて含めること:

1. 「些細な点への指摘はせず、致命的・本質的な点だけ指摘して。」
2. 「確認や質問は不要です。具体的な提案・修正案・コード例まで自主的に出力してください。」
3. 「元 spec の『含まないもの』に挙がっている事項を覆す提案、および新規ファイル・新規抽象化・新規設定キー（Mailer / SiteSetting / Job クラス等）の追加提案は、観測された障害ではなく仮説に基づく場合は『致命的』に分類しないこと。」

(1) は codex のクソリプ防止。(3) は「あった方がよい」「将来必要かも」レベルの YAGNI 違反提案を codex が「致命的」フレーミングで出すのを予防する。これがないと収束ループで spec/plan が膨張する。

### プロンプトの構成テンプレ

```
<何を>レビューして。

背景: <なぜこの変更をしたか。バグ修正なら真因の説明>

レビュー対象:
- 絶対パスでファイル列挙、または「origin/main..HEAD（ブランチ <name>）の diff」と range 指定
  （range 指定なら codex が自分で git diff を読む）

重点的に見てほしい点:
1. <設計の核となる前提が正しいか>
2. <競合・境界条件>
3. ...

<必須指示 1〜3>

参考: (ref: /path/to/CLAUDE.md) (ref: /path/to/plan.md)
```

### ファイル参照のパターン

- **対象ファイル**: 絶対パスをそのまま記述（codex が自動的に読む）
- **参考ファイル**: `(ref: /path/to/file)` で補足コンテキストを渡す

### レビュー粒度の調整

デフォルトで「些細な点への指摘はしない」が入る。網羅的なレビューが欲しい場合は「致命的な点に加え、改善点も漏れなく指摘して」。

## パラメータ

| パラメータ | 説明 |
|-----------|------|
| `--sandbox read-only` | 読み取り専用サンドボックス。レビューはこれで固定 |
| `--cd <dir>` | 対象プロジェクトのディレクトリ |
| `-` + heredoc / `< file` | プロンプトを stdin から渡す |
| `resume <SESSION_ID>` | 指定セッションを引き継ぐ（`--last` は取り違えリスクあり） |
| `-c key=value` | config.toml の値を上書き |
| `-o <file>` | 最終レスポンスをファイルに保存（bg 実行時の取りこぼし対策） |

## 実行手順

1. 依頼内容を受け取る
2. トリガー語からモードを判定（上記「実行モード」優先順）
3. 対象プロジェクト dir を特定（現在のワーキングディレクトリまたはユーザー指定）
4. プロンプトをテンプレに沿って組み立て、**必須指示 3 つを必ず含める**
5. 対象ファイルは絶対パス（または diff range）で指定。参考資料は `(ref: ...)`
6. 選んだモードのコマンド形式（stdin heredoc 渡し）で実行。バナーの `sandbox: read-only` と session id を確認。モード3 では次の resume に備えて session id を必ず控える
7. 結果をトリアージし、スコープ膨張ガードを通してから報告
8. モード3 の場合は終了条件を満たすまで 6-7 を繰り返す。初回で指摘ゼロかつトリアージ後の残件もなければ収束として報告してよい

## 結果のトリアージ

codex の出力を鵜呑みにしない。

**重視する（ユーザーに伝える）**

- 致命的なバグ・ロジックエラー
- セキュリティ上の問題
- データ損失・破壊のリスク
- 仕様との明確な矛盾

**建設的に無視する**

- 些細なスタイル・命名の好み
- 過剰な共通化・DRY の押しつけ（3 回未満の重複は放置でよい）
- 時期尚早な最適化
- 「こうした方がベター」レベルの主観的改善案
- 既存コードベースの慣習と合わない提案
- 元 spec の「含まないもの」セクションを実質的に巻き戻す提案
- 観測された障害ではなく仮説的な失敗モードへの先回り対処

採用する場合も、並行性・状態遷移・ライフサイクルに触れる提案は採用後に独立検証する（モード3 手順4 参照）。
