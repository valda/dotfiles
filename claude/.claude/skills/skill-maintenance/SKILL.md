# skill-maintenance

`~/.claude/skills/*/SKILL.md` を編集するときの方針。新規作成は `skill-creator` を使う。ここは既存 skill の手入れ専用。

## 編集方針

### 1. 時系列ログを残さない

skill は「今の最良解」を書く場所。試行錯誤の履歴は git log に任せる。

**削除対象**:

- 日付付き観測ログ (`2026-06-15 に確認`, `yyyy/mm/dd 観測` 等)
- PR 番号・issue 番号への参照（`#123 で修正済み` 等）
- 「以前は X だったが今は Y」「旧バージョンでは…」型の経緯
- 「実機で確認した」「動作確認済み」等の鮮度メタ情報
- 重要・必須・注意・⚠️ 等の強調ラベルの濫用

**残すもの**:

- 最新の解決方法（手順・コマンド・プロンプト本文）
- pitfall（踏み抜きやすい落とし穴）とその回避策
- 採否の理由が将来判断に効く場合のみ、根拠を 1 行

判断基準: その行を消したとき、未来の自分が同じ罠を踏むか？ Yes なら残す。No なら消す。

### 2. プロンプトをリーンに保つ

- 同じことを 2 回書かない。節をまたいで重複したら統合する。
- 「〜してください」「〜することが推奨されます」等のフィラーを削る。命令形で短く。
- 箇条書きの各項目は 1 行を目標に。2 行を超えるなら節に切り出すか削る。
- 例示は 1 個に絞る。複数バリエーションを並べない（最良の 1 個＋ pitfall に集約）。
- セクション見出しは内容を予測できる名詞句に。動詞句や疑問形は避ける。

## 作業手順

1. **対象 skill を Read**: SKILL.md 全文を読む。長ければ分割読みでなく一気に把握する。
2. **時系列要素を抽出**: 日付・PR 番号・「以前は」「最近」等を grep 的に拾う。
3. **重複・冗長を抽出**: 同じ警告が複数節にある／同じ手順が再掲されている箇所を拾う。
4. **編集案を提示**: diff 形式または箇条書きで「削除・統合・短縮」を列挙。ユーザー確認後に Edit。
5. **収束チェック**: 編集後の SKILL.md を再度読み、行数と構造を確認。frontmatter の `description` も最新の本文と整合するか見る。

複数 skill を一括メンテする場合は skill 単位で順に処理する。並行編集は差分管理が破綻するので避ける。

## 触らないもの

- `scripts/` 配下の実行コード（skill の動作そのもの）
- frontmatter の `name`（slug 変更は skill 参照を壊す）
- ユーザーが明示的に「履歴として残す」と指定した節

## pitfall

- **トリガー語句を削りすぎない**: `description` のトリガー一覧は呼び出し精度に直結する。冗長さの整理対象から外す。
- **pitfall 自体をログ扱いしない**: 「過去に踏んだ罠だから消す」は誤り。再発防止に効くものは保持。
- **強調ラベルをゼロにしない**: 本当に致命的な 1〜2 箇所だけ残す。全部消すと優先度の山が平坦化する。
