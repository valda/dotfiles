## あなたのタスク

$ARGUMENTS の日時以降、あるいは期間のコミットログから進捗報告を作成して下さい。
$ARGUMENTS がなければ本日、あるいは直近の一日から作成してください。

## 進捗報告のガイドライン

- フォーマットはプレーンテキスト形式。
- git log の取得では `--oneline` を付けずコミットメッセージの詳細まで確認する。ただしソースコードの差分まで確認する必要はありません。
- 規模の大きなコミットは、変更の影響度の大きさ順にまとめる。ただし、レポート中に "最高影響度" のような評価結果は出力しない。
- 些細な修正は簡潔にまとめる。
- 上記以外はコミットの時系列でまとめる。
- レポート中に "完全実装" や "完全解消" といった誇大なレポートは行わない。
- 実装の詳細（クラス名やソースファイル名）に立ち入った表現は控える。"◯◯の機能に対する□□の追加"といったような外形的な表現を心がける。

## 出力テンプレート

以下の構造に従って出力する。インデントは半角スペース2つ。

```
{プロジェクト名}

進捗報告（{対象期間}）
対象期間のコミット数: {N}件

---
{テーマ見出し1}

{サブ見出し（任意）}

- {変更の要約}
- {変更の要約}
  - {補足詳細}

{サブ見出し（任意）}

- {変更の要約}

---
{テーマ見出し2}

- {変更の要約}

---
{必要に応じてテーマ見出しを追加}
```

### テンプレート運用ルール

- **ヘッダー**: プロジェクト名 → タイトル行 → 対象期間・コミット数の順。コミット数が少ない場合は変更ファイル数・行数を添えてもよい
- **セクション区切り**: テーマ間は `---` で区切る
- **テーマ見出し**: 関連するコミットを意味のあるまとまりでグルーピングし、見出しを付ける
- **サブ見出し**: テーマ内でさらに分類が必要な場合のみ使用（任意）
- **項目の書式**: 各項目はハイフン箇条書き。補足はインデントして子箇条書き
- **少数コミットの場合**: 番号付きリスト（`1.` `2.`）で時系列順に並べ、各項目にコミット時刻を添えてもよい
- **まとめ**: 期間の変更が多い場合は省略。少ない場合は末尾に1行の総括を添えてもよい（任意）
