## 概要

`docs/todo/` ツリーで管理されるタスク・ナレッジを、セッションのライフサイクルに沿って読み書きする統合スキル。

- **check**: セッション開始時。索引と選択コンテキストを読み込み、今回の作業目標を決める。
- **sync**: 作業中・終了時。進捗とナレッジを記録し、索引を最新化する。

## モード選択

1. 引数 `check` / `sync` が明示されていればそれに従う。
2. 引数なしの場合、ユーザー発話のトリガー語句から推定:
   - 「開始」「確認」「今日の作業」「check」系 → **check**
   - 「終了」「更新」「記録」「同期」「sync」系 → **sync**
3. 推定不能ならユーザーに確認。

## 共通事項

### ファイル構成

```
docs/todo/
├── index.md                       # 軽量索引（コンテキスト別サマリ）
├── contexts/
│   └── [コンテキスト名].md        # プロジェクトごとに動的命名
├── knowledge/
│   ├── [コンテキスト名]-learnings.md
│   └── debug-patterns.md          # 汎用デバッグパターン
└── archive/YYYY-MM/               # 完了タスクのアーカイブ
```

コンテキスト名はプロジェクトに応じて動的に決まる（例: `webapp`, `infrastructure`, `data-pipeline`, `ml-training` など）。

### コンテキストの動的検出

`docs/todo/contexts/*.md` を列挙して既存コンテキストを把握する。新規が必要なら名前をユーザーに確認してから作成。

### 出力規約

- すべてのファイル内容・対話メッセージは日本語。
- 重要度や緊急度は本文中で明示（「緊急」「重要」など）。
- タイムスタンプは `YYYY-MM-DD` または `YYYY-MM-DD HH:MM`。

---

## check モードの手順

セッション開始時。必要最小限の読み込みで作業目標を決定する。

1. **ディレクトリ確認**
   `docs/todo/` が無ければ「sync モードで初期化できる」旨を案内して終了。

2. **索引の読み込み**
   `docs/todo/index.md` を読み、各コンテキストの状況サマリを提示。索引が無ければ「sync モードで作成」と案内。

3. **コンテキスト選択**
   `contexts/` を列挙し、ユーザーに次の選択肢を提示:
   - 既存コンテキスト名のいずれか
   - `new` 新規作成
   - `all` 全コンテキストの概要表示

4. **詳細の読み込み**
   選択コンテキストについて以下のみを読む（他コンテキストの大きいファイルは読まない）:
   - `contexts/[name].md`
   - `knowledge/[name]-learnings.md`（直近の学習履歴）
   - `knowledge/debug-patterns.md`（該当領域にマッチするパターンのみ抽出）

5. **作業目標の決定**
   タスク群から今回のセッションの目標を確定:
   - 最優先課題 / 継続タスク / 新規着手のいずれか
   - 予想時間と期待成果物
   - 関連する過去の失敗例

6. **準備完了の通知**
   選択コンテキスト・目標・関連ナレッジの読み込み状況を要約し、セッション終了時に `sync` を実行する旨を伝える。

---

## sync モードの手順

作業中の節目またはセッション終了時。進捗と知見を記録する。

1. **コンテキスト確認**
   今回作業したコンテキストをユーザーに確認。`contexts/*.md` を列挙して提示。該当が無ければ新規作成フローへ。

2. **ディレクトリ整備**
   不足分のみ作成:
   ```
   docs/todo/{contexts,knowledge,archive/YYYY-MM}
   ```

3. **新規コンテキスト作成**（必要時）
   名前をユーザーに確認し、以下のテンプレートで作成:

   `contexts/[name].md`:
   ```markdown
   # [コンテキスト名] Tasks

   作成日: YYYY-MM-DD
   最終更新: YYYY-MM-DD

   ## 緊急タスク
   ## 重要タスク
   ## 一般タスク
   ## 完了したタスク
   ## メモ・注意事項
   ```

   `knowledge/[name]-learnings.md`:
   ```markdown
   # [コンテキスト名] 学習記録

   作成日: YYYY-MM-DD

   ## 学習履歴
   ```

4. **タスク更新（対話で確認）**
   - 完了タスク: チェック / 取り消し線 / 必要なら `archive/YYYY-MM/` に移動
   - 新規課題: 優先度（高/中/低）を付けて該当コンテキストへ追加
   - 進行中: 状況を追記
   - コンテキスト跨ぎ: 移動先を確認して反映

5. **ナレッジ記録**

   **コンテキスト別** `knowledge/[name]-learnings.md` に追記:
   ```markdown
   ## YYYY-MM-DD [コンテキスト名] セッション学習記録

   ### 成功したアプローチ
   ### 失敗したアプローチ
   ### 重要な発見・気づき
   ```

   **汎用デバッグ** `knowledge/debug-patterns.md` に追記（必要時）:
   ```markdown
   ## YYYY-MM-DD [コンテキスト] デバッグログ

   ### 問題: [概要]
   **症状**:
   **調査プロセス**: 1. 仮説 → 結果 / 2. … / 3. 最終解決
   **学んだこと**:
   **関連コンテキスト**:
   ```

   ナレッジ抽出のための質問例:
   - 今回のセッションで学んだ重要なことは？
   - 失敗・避けるべきアプローチは？
   - デバッグで時間を要した問題は？
   - 他コンテキストにも応用可能な知見は？

6. **索引の更新**
   `index.md` をコンテキスト別サマリ形式で書き換え:
   ```markdown
   # プロジェクト TODO 索引

   最終更新: YYYY-MM-DD HH:MM

   ## コンテキスト別状況サマリー

   ### [name] (緊急度: 高/中/低)
   - アクティブ: X 件
   - 完了待ち: Y 件
   - 最新の課題: [1 行]

   ## 緊急対応が必要な項目
   ## 今週の目標
   ```

---

## 注意事項

- セッション開始時は **check**、終了前は **sync** を必ず実行する流れを促す。
- 索引（`index.md`）と詳細（`contexts/*.md`, `knowledge/*.md`）を分離することで、開始時の読み込みを軽量に保つ。
- 小さな学び・失敗も記録する。将来セッションのナレッジ資産になる。
- コンテキスト名・粒度はプロジェクト固有に最適化してよい。汎用名を強制しない。
- 跨コンテキストの知見は複数ファイルへの記録を検討。
